## 已修复问题

### P0：源码硬编码密钥

**问题**：`server.py` / `download_history.py` 里存在硬编码 Tushare token。  
**风险**：密钥泄露、代码仓库污染、部署环境不可迁移。  
**修复**：新增 `core/config.py`，通过环境变量 `TUSHARE_TOKEN` 懒加载 Tushare client。服务端通过 systemd user drop-in 注入运行环境。  
**验证**：源码 grep 已确认不再存在旧 token literal 或 `TUSHARE_TOKEN =`。

### P0：宏观数据存在前视偏差风险

**问题**：CPI/PMI/M2/社融等月度宏观数据容易把统计期月份误当成可交易可见日期。  
**风险**：回测和信号会提前使用未来才公布的数据。  
**修复**：新增 `core/macro_calendar.py`，对月度宏观数据补 `release_date` 并按 `as_of` 过滤。  
**说明**：当前 release date 是保守规则推断；后续最好接入真实公告发布日期。

### P1：因子输出结构不统一

**问题**：不同因子输出字段分散，部分没有稳定 `factor_value` / `signal_strength` / `trigger` / `meta`。  
**风险**：前端、推送、IC 监控和外部工作流消费困难。  
**修复**：

- `BaseFactor._make_signal()` 统一补齐 `strength`、`signal_strength`、`confidence`、`trigger`、`factor_value`、`meta`。
- `core/factor_runner.py` 在执行层统一补齐/规范 `factor_data.factor_value`，保留旧字段。
- 宏观因子输出补 `period_date`、`release_date`、`factor_value`。

### P1：`server.py` 业务逻辑过重

**问题**：路由层同时承担因子导入、实例化、执行、信号记录、IC 快照、数据刷新任务、调度器启动等职责。  
**风险**：后续扩展因子或调试 API 时容易互相影响。  
**修复**：新增 `FactorRunner`，把因子执行核心逻辑迁出 `server.py`；新增 `core/data_refresh.py` 承载国内/外盘数据刷新；新增 `core/scheduler.py` 承载 APScheduler 启动和 gunicorn 多 worker 文件锁。`server.py` 保留兼容包装函数、API 路由和少量任务入口。

### P1：`forex` 因子列名不兼容

**问题**：实际 `usd_cny` 数据含 `DEXCHUS` 列，旧代码只兼容 `close/value`，导致宏观链报 `'value'`。  
**修复**：`factors/macro/forex.py` 兼容 `close/value/DEXCHUS/usd_cny/USD_CNY`，并补 `factor_value`。

### P2：pandas 统计 warning

**问题**：`series.pct_change().std()` 在含 NaN/inf 的序列上触发 `RuntimeWarning: invalid value encountered in subtract`。  
**修复**：在 `BaseFactor` 的 `_zscore()`、`_adaptive_threshold()`、`_adaptive_zscore_threshold()` 里统一清洗非数值、NaN、inf。

### P2：scheduler 文件锁句柄警告

**问题**：Gunicorn 多 worker 下 scheduler 文件锁句柄未保留，严格 warning 下出现 `ResourceWarning`。  
**修复**：锁 owner worker 将文件句柄保存到全局变量，未持锁 worker 主动关闭句柄。

### P2：推送价格分位口径错误

**问题**：`core/push.py` 中 `percentile` 已是“比当前更便宜的交易日占比”，展示时误做 `100 - percentile`，导致高低区间反转。  
**修复**：直接使用 `percentile`，并整理为 `_get_close_prices()`、`_period_label()`、`_position_label()` 等小函数。

## 金融逻辑深审补充（本轮未直接改交易逻辑）

> 原则：本节只记录金融逻辑、统计口径、回测可靠性风险；不在没有回测证据的情况下直接改阈值、方向或持仓天数。

### A. 数据口径与回测可信度

#### A1. 主力连续换月调整存在误判风险

**位置**：`core/data_bus.py` 的 `_adjust_roll_gap()`。

**观察**：当前用“单日收益绝对值 > 20 日滚动波动率 × 5”识别换月跳空，并把后续价格整体平移。这个方法能消除明显断点，但也可能把真实行情跳涨/跳跌误识别为换月，尤其是商品在政策、库存、地缘冲突、天气冲击下出现真实跳空时。

**影响**：

- 均值回归类因子的 z-score、percentile 可能被平滑过度。
- 动量、波动率、期限结构类因子的真实风险可能被低估。
- `close_raw` 已保留，但多数因子默认使用调整后的 `close`，需要明确这是“回测收益口径”还是“信号平滑口径”。

**建议**：

1. 优先接入真实主力合约切换日期或合约代码字段。
2. 将 `close_raw`、`close_adj`、`return_raw`、`return_adj` 显式区分。
3. 对动量/波动率/止损类因子使用 raw return 或真实可交易连续收益；对估值/位置类因子可使用调整后价格。

#### A2. IC 计算目前只做同日交集，缺少交易日可用性对齐

**位置**：`evaluation/ic_monitor.py` 的 `compute_ic()`。

**观察**：当前 `snapshot_date` 与 `forward_returns` 用日期 index 直接 intersection。对宏观数据来说，虽然因子里补了 `release_date/as_of`，但 IC 快照是否严格按“信号生成日”落库仍取决于快照写入时点；如果历史重算时 snapshot_date 不是 release_date 之后的交易日，仍可能有残余时点偏差。

**建议**：

1. IC 快照表增加 `available_date` 或使用 `release_date` 作为宏观因子可用日。
2. 对非交易日发布日期，映射到下一交易日。
3. IC 评估区分：截面 IC、时间序列 IC、单资产方向命中率，避免把不同类型因子混用一个口径。

#### A3. 宏观 release date 是保守推断，不是真实公告日

**位置**：`core/macro_calendar.py`。

**观察**：CPI/PMI/M2/社融使用固定 business-day lag。这个比直接用统计期日期安全很多，但仍不是精确公告日。PMI 通常月末/次月初，社融发布时间不稳定，节假日影响明显。

**建议**：后续数据层保存真实 `release_date`，并对节假日、周末、夜间发布时间映射到下一可交易时点。

---

### B. 信号聚合与仓位解释

#### B1. 聚合器的信心度容易被“信号数量”和“强弱权重”混淆

**位置**：`core/signal_aggregator.py`。

**观察**：加权聚合中 `weight = abs(strength) * confidence`，方向由净权重决定，最终 `confidence = total_weight / len(signals)`。这会把“信号数量多”与“独立证据强”混在一起；如果多个高度相似因子同时触发，即使做了 trigger group 去重，也不等于统计独立。

**风险**：组合层可能过度自信，尤其是宏观链、商品链中多个因子共享同一底层价格或同一宏观变量。

**建议**：

1. confidence 改为“证据质量分”，position_size 另算。
2. 相关性去重从 trigger 配置升级为基于历史收益/因子值滚动相关矩阵。
3. 聚合结果输出分解：方向票数、净暴露、平均置信、相关性折扣、冲突度。

#### B2. 止损参数目前像展示字段，不像真实风控规则

**观察**：各因子的 `stop_loss` 是静态百分比，未结合品种波动率、合约乘数、保证金、滑点、涨跌停、夜盘跳空。

**建议**：

- 商品期货统一引入 ATR/历史波动率止损。
- 根据 `holding_days` 和 `volatility_20d` 计算风险预算。
- 区分“信号失效条件”和“交易止损条件”。

### C. 具体因子逻辑风险点

#### C1. PMI → 金属因子未做可用日过滤

**位置**：`factors/cross/pmi_metals.py`。

**观察**：宏观单因子 `pmi.py` 已接入 `available_asof()`，但跨品种传导因子 `pmi_metals.py` 直接 `self.load("pmi")`，没有按 `as_of/release_date` 过滤。

**风险**：跨链回测时可能继续使用未公布 PMI。

**建议**：所有使用宏观数据的 cross/system 因子也统一走 `available_asof()`。

#### C2. 社融同比计算可能受月度缺口和春节错位影响

**位置**：`factors/macro/social_financing.py`。

**观察**：当前用最近 3 个月均值对去年对应附近月份均值估算 `sf_growth`。这比单月同比稳健，但社融受春节错位、政策窗口、政府债发行节奏影响很大。

**风险**：阈值 `>12`、`>10`、`<8` 在不同信用周期下含义不同，可能产生 regime drift。

**建议**：

1. 优先使用官方“社融存量同比”而非增量滚动估算。
2. 加入春节错位处理或 3M/6M 同比双口径。
3. 阈值改为历史分位或滚动 z-score，而非固定绝对值。

#### C3. 动量因子分数非线性较强，可能过快饱和

**位置**：`factors/technical/momentum.py`。

**观察**：`tanh(m5*30)`、`tanh(m20*15)`、`tanh(acceleration*50)` 对 2%-5% 的价格波动已经很敏感，容易快速接近 ±1。

**风险**：不同波动率品种之间不可比；高波动品种天然更容易触发强动量。

**建议**：使用波动率标准化收益（return / realized_vol）或按品种历史分位归一化。

#### C4. 波动率因子本身输出 HOLD，但 `signal_strength()` 仍给方向性分值

**位置**：`factors/technical/volatility.py`。

**观察**：`signal()` 明确说方向不确定，返回 HOLD；但 `signal_strength()` 在高波动时为负、低波动时为正。若执行层或聚合层使用 `signal_strength()` 作为方向强度，可能引入隐性方向偏见。

**建议**：

- 波动率因子只输出 risk_modifier / position_multiplier。
- 不直接贡献 BUY/SELL 方向。
- 如要表达“低波动突破”，必须结合动量方向。

#### C5. 期限结构因子需要确认近远月数据是否同日同合约族

**位置**：`factors/technical/term_structure.py`。

**观察**：当前按 `date` merge 近远月 close。若近月/远月文件不是严格同一交易日、同一品种合约族，spread percentile 会失真。

**建议**：数据层保存合约代码、到期月份、交易所、主力/次主力标识；期限结构使用同日同品种合约链构建，而不是两个独立 parquet 简单相减。

#### C6. 油金传导、黄金因子、汇率商品因子存在宏观解释重叠

**位置**：`factors/cross/oil_gold_link.py`、`factors/metals/metals.py` 的 GoldFactor、`factors/cross/forex_commodity.py`。

**观察**：多个因子都会把油价、汇率、通胀预期解释为黄金/商品方向信号。这些逻辑有经济含义，但底层驱动高度相关，且某些情景下方向并不稳定：

- 油涨可能是需求强，也可能是供给冲击。
- 黄金对油价的响应取决于实际利率、美元、风险偏好。
- 人民币贬值对进口商品是成本推升，但也可能对应国内需求走弱或美元走强压制外盘商品。

**建议**：加入 regime 条件，例如美元指数/实际利率/权益风险偏好/库存状态；并在聚合层对共同宏观驱动做去重。

#### C7. 黄金 `signal_strength()` 与部分 signal 方向可能不完全一致

**位置**：`factors/metals/metals.py` 的 GoldFactor。

**观察**：`signal_strength()` 以 `zscore / 3` 为基础，zscore 越高强度越正；但 `signal()` 中 zscore 高且波动高会触发 SELL（过热回调），zscore 极低触发 BUY。若外部调用 `signal_strength()` 不看 `signal()`，方向解释会冲突。

**建议**：均值回归类因子的 strength 应与交易方向一致，或拆成 `valuation_score` 与 `trade_signal_strength`。

### D. 下一步建议优先级

1. **先做数据口径表**：每个 parquet 的来源、频率、时区、是否复权/换月、可交易时间、字段含义。
2. **统一宏观 as_of**：所有 cross 因子中引用宏观数据的地方都接入 `available_asof()`。
3. **重构技术因子输出**：把方向信号、风险过滤器、仓位修正器分开，避免 HOLD 因子贡献方向。
4. **建立最小回测框架**：至少输出每个 trigger 的触发次数、胜率、平均收益、最大回撤、IC、分 regime 表现。
5. **阈值再校准**：只有在 trigger 级回测后，才改 confidence/strength/holding_days/stop_loss。
