**审查工具**：Claude Code (FreeModel) + OpenClaw  
**因子总数**：39 个，53 条分析链

## 一、金融角度审查

### 1.1 因子计算逻辑 ✅ 总体合理

| 因子 | 经济逻辑 | 信号方向 | 风险等级 |
|------|----------|----------|----------|
| **猪粮比** | 收储/抛储政策预期反向交易 | ✅ 正确 | 低 |
| **动量** | 三周期动量+加速度+波动率过滤 | ✅ 正确 | 低 |
| **油金联动** | 油→通胀预期→实际利率→金 | ⚠️ 只有BUY | 中 |
| **饲料成本** | 饲料占养殖成本60-70% | ✅ 正确 | 低 |
| **压榨利润** | 利润→开工率→豆粕供给 | ✅ 正确 | 低 |
| **铜金比** | 风险偏好代理指标 | ✅ 正确 | 低 |
| **猪鸡价差** | 猪贵→鸡肉替代需求 | ⚠️ 数据缺失 | 高 |
| **期限结构** | 近远月价差→供需格局 | ✅ 正确 | 低 |
| **PMI** | PMI方向→经济周期→股市 | ✅ 正确 | 低 |
| **社融** | 社融增速+剪刀差 | ✅ 正确 | 低 |
| **CPI** | 通胀→货币政策预期 | ✅ 正确 | 低 |
| **季节性** | 历史同期月度收益统计 | ✅ 正确 | 低 |
| **波动率** | 短长期波动率比值 | ✅ 正确（只发HOLD） | 低 |
| **商品→权益** | 商品估值+成本+权益动量 | ✅ 正确 | 低 |

### 1.2 关键风险点

#### 🔴 高风险

1. **猪鸡价差因子数据缺失**
   - `chicken_spot` 尚未接入稳定数据源
   - `pig_chicken_ratio` 无法计算，因子实际不可用
   - **建议**：尽快接入鸡肉现货数据，或标记因子为"实验性"

#### 🟡 中风险

2. **油金联动因子只有多头方向**
   - 无论油价大涨、暴跌还是高相关性同涨，都触发 BUY 黄金
   - 缺少 SELL 信号，不对称
   - **建议**：补充"油价稳定+金价过热→SELL"的触发条件

3. **宏观数据发布日期是估算值**
   - `RELEASE_LAG_DAYS` 是固定延迟，不是真实发布时点
   - 遇到延迟发布（节假日、统计口径调整）会偏乐观
   - **建议**：接入真实发布日历，或用数据入库时间作为兜底

4. **前视偏差风险（修订值）**
   - AKShare 给的是最新修订后的序列，回测时用"未来才修订出来的数字"
   - **建议**：做 point-in-time 快照，保存每次抓取的版本

5. **铜金比资产标签不一致**
   - registry 上写的是沪深300ETF，但实际信号约 2/3 概率指向黄金
   - **建议**：统一资产标签，或按信号方向动态切换

#### 🟢 低风险

6. **季节性因子样本量要求偏低**
   - 要求 ≥3 年历史数据，但某些品种可能只有 3-5 年数据
   - **建议**：提高到 ≥5 年，或增加样本量不足时的置信度衰减

7. **压榨利润公式忽略基差**
   - 用期货价格计算利润，忽略基差和副产品价格波动
   - **建议**：在文档中注明局限性，或增加基差修正项

## 二、代码角度审查

### 2.1 架构设计 ✅ 总体良好

- **BaseFactor 接口**：关注点分离清晰，自适应阈值优先
- **DataBus 单例**：统一缓存数据避免重复 I/O
- **FactorRunner**：驱动因子执行，支持链式调用
- **SignalAggregator**：两层去重（相关性+宏观驱动），三种融合方法
- **ICMonitor**：带状态的因子衰减监控

### 2.2 关键问题

#### 🔴 高风险

1. **BaseFactor 缓存失效**
   - `_get_or_calculate` 方法定义了缓存逻辑但 `calculate()` 调用后没有回填 `_cached_data`
   - 缓存实际上是失效的
   - **建议**：补上回填逻辑

#### 🟡 中风险

2. **`_make_signal` 信息重复**
   - `kwargs` 同时存 flat 字段和 `meta`，同一份信息存两次
   - **建议**：规划弃用路径，逐步迁移到 `meta`

3. **`_threshold_cache` 复用率低**
   - 用 `len(series)` 做 key，新增一行数据就 miss
   - **建议**：改用更稳定的 key（如日期范围）

4. **RSI 计算效率低**
   - 用 Python for 循环逐点更新平均收益/损失
   - **建议**：改用 Wilder 平滑的 `ewm(alpha=1/period)`

5. **SignalAggregator 相关性去重是静态的**
   - `compute_signal_correlation_discount` 当前空实现
   - **建议**：实现基于历史相关矩阵的动态折扣

#### 🟢 低风险

6. **server.py 过重**
   - 所有 API 路由和业务逻辑混在一起
   - **建议**：拆到 `services/` 或 `orchestrator/`

7. **测试覆盖不足**
   - 缺少对所有因子最小样本的测试
   - 缺少信号触发逻辑的测试
   - **建议**：扩展测试覆盖

## 三、API 设计审查

### 3.1 API 端点清单

| 类别 | 端点 | 功能 |
|------|------|------|
| 健康检查 | `GET /health` | 服务存活检查 |
| 链条分析 | `GET /analyze/<chain>` | 执行单链/综合链分析 |
| 因子 | `GET /factor/<chain>` | 仅返回因子值 |
| 信号 | `GET /signal/<chain>` | 仅返回信号 |
| 建议 | `GET /recommend/<chain>` | BUY/SELL/HOLD 建议 |
| 每日总览 | `GET /recommendations/daily` | 所有链条分组 |
| 信号历史 | `GET /signals/history` | 查询历史信号 |
| IC 监控 | `GET /ic/<factor>` | 单因子 IC 分析 |
| 回测 | `GET /trigger_backtest` | 触发器回测 |
| 推送 | `GET /push/<chain>` | 手动触发推送 |

### 3.2 安全问题

1. **无认证机制** — 所有 API 端点公开可访问
2. **无速率限制** — 容易被滥用
3. **敏感信息暴露** — 信号历史和 IC 数据可能包含交易策略

**建议**：对于个人系统，至少加一个 API Key 认证

## 六、总结

这个量化分析系统**整体设计良好**，金融逻辑合理，代码架构清晰。主要风险在于：

1. **数据层面**：猪鸡价差缺数据、宏观数据有前视偏差风险
2. **代码层面**：缓存失效、信息重复、效率问题
3. **安全层面**：无认证、无速率限制

对于一个**个人量化分析系统**来说，当前的完成度已经相当高。建议按优先级逐步修复上述问题，重点关注 P0 级别的三个问题。
